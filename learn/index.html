Building with Fused Render

A project is just a folder with a page and a Python file. The fastest way to make one is to ask Claude Code — it already knows the Fused Render layout. Here's the prompt.

Prerequisites

This assumes Claude Code and the Fused Render skills are installed — the Claude Code skills page has the one-line install and a live check of your machine.

✦ Paste this into Claude Code
Create a new Fused Render project that shows my computer's CPU and memory usage as live gauges, refreshing every second.

Swap in your own idea after “…that”: a treemap of my Downloads folder, a searchable table from a CSV I pick, a world clock for three time zones. No need to mention HTML or Python — Claude fills that in.

The mental model

A project is just a folder with two files. The HTML is what you see; the Python is the work. That's the whole structure:

📁 my_tool/
├── 📄 index.html → what you see — the UI, in HTML & JS
└── 🐍 tool.py → the work — a main() that returns JSON
HTML  ⇄  Python — they talk over the fused bridge injected into every page
▸ Click index.html and it renders as a live page. Edit either file and the page reloads itself.

HTML calls Python

Next to a page sits its Python. The page sends input to it and renders what comes back — try it: this calls hello.py for real when opened in the app.

Python hasn't said anything yet.
# hello.py — the whole contract
def main(name: str = "world"):
    return {"greeting": f"Hello, {name}!"}

The page's side is one line: await fused.runPython("./hello.py", {name}). A function named main() that returns JSON is all it takes.

State lives in the URL

Controls don't keep private state — they write to the page's URL through fused.params. Drag the slider and watch the address bar: the ?level= value updates live. Refresh, bookmark, or share that link and the page comes back exactly as you left it.

The slider writes fused.params.set("level", …), which appends ?level= to the URL. Nothing is stored in hidden JS state — the URL is the state, so it survives a refresh and travels in a shared link.

The page's address bar tracks it live:

127.0.0.1:1777/view/…/my_tool/index.html?level=40

The fused API

Everything your page can call. It's injected automatically — never <script src> anything for it. On the Python side, you only ever write a function named main().

await fused.runPython(path, params)Runs main(**params) of the .py at path (relative to the page) and resolves with its return value. Params are strings; scrubbing a slider auto-cancels the runs it moves past.
fused.params.get(k)Read a URL param (a string, or undefined). The URL is your state.
fused.params.set(k, v)Write a URL param — v must be a string, so String(n) yourself. Then fires onChange.
fused.params.onChange(cb)Run cb after every applied set — the single place you re-render from.
await fused.readFile(path)File contents as text — no Python needed when you just want the bytes.
await fused.writeFile(path, text)Write text atomically (optional mtime lock to avoid clobbering).
await fused.stat(path)Metadata {size, mtime, is_dir, …} — size-guard before reading big files.
fused.rawUrl(path)A URL for the raw bytes — use it as the src of an <img>, <video>, or a download link.
Good to know

Each runPython call runs main() in a fresh subprocess that's killed at 30 seconds. For a genuinely long job, write the result to a file from a separate script and have the page read the finished file.

Supported Python libraries

Your .py files run in the app's bundled Python (3.12), which ships the standard library plus this data stack — no pip install needed (or possible):

Datanumpy pandas polars pyarrow duckdb scipy openpyxl msgpack
Geospatialshapely geopandas rasterio zarr
Plots & imagesmatplotlib pillow
Documentspymupdf pikepdf fpdf2 python-pptx
Network & cloudrequests httpx botocore google-auth
Logsdrain3

Stick to these (and the standard library) when writing Python for your pages — imports outside this set will fail in the packaged app.

Workflow

Claude Code skills

Fused Render ships a set of Claude Code skills — packaged instructions that teach the claude CLI how this project works. Install them once and Claude authors views, drives the explorer, and registers templates fluently, without you spelling out the conventions each time.

Prerequisite

Skills are a feature of Claude Code, so the claude CLI must be installed on your machine and on your PATH. The skills add fused-render know-how on top; they don't replace Claude Code itself.

Checking your machine for the claude command…

1Install the plugin

The skills are distributed as a Claude Code plugin from this repo's marketplace. From inside Claude Code, add the marketplace, then install the plugin:

/plugin marketplace add fusedio/fused-render
/plugin install fused-render@fused-render

Run /plugin afterwards to confirm fused-render is installed. The skills then load automatically in any session — including when Claude Code is driven from inside the app's Claude panel.

Working inside the repo?

If you're developing fused-render itself, the skills are already present as ./skills/ in the checkout — no install needed. Maintainer-only skills (dev-env setup, cutting a release) live separately under .claude/skills/ and never ship in the plugin.

2What you get

The plugin installs three skills that cover the whole surface. You never name them — Claude reads their descriptions and picks the right one for what you ask.

fused-render-usage

Running and using a project — start the local server, browse the filesystem, and open views or files by URL (embed vs full-shell modes).

fused-render-authoring

Writing and debugging views — the fused.runPython() bridge to local Python, the main() contract, URL-synced params, and the file-IO helpers (readFile/writeFile/stat/rawUrl).

fused-render-custom-templates

Registering your own preview template for a file extension under ~/.fused-render/templates/ — so opening that file type renders your page (or offers it as a switchable mode). Pairs with the Templates concepts page.

3Just ask

You describe the outcome in plain language; Claude loads the matching skill and follows its conventions. No need to mention HTML, Python, or the fused API.

Build me a Fused Render view that reads the CSV I'm looking at, with a slider to filter rows by value, and keep the filter in the URL.

This is the same promise behind the “hand it to Claude Code” prompt on the Building with Fused Render page — the skills are what make “Claude knows the Fused Render layout” true.

Concepts

Templates

A template decides how a file renders. HTML files are their own template; every other file type opens through a template that turns it into a page.

1Core vs custom templates

Core templates ship with Fused Render and cover common file types out of the box. Custom templates are ones you write — replace a built-in, or bind your own to a file type or pattern.

Core

Built in, always available. Sensible defaults for the file types you open most.

Custom

Your own HTML template, bound to a type or path. Let Claude read the existing templates to build on top of them.

The Templates page (bottom-left of the app) is where these bindings live — every file type maps to an ordered list of templates, and the first is the default.

3Right-click to open

Right-click a file → Open with FusedRender renders it through the matching template. The OS-level menu entry is registered by the packaged macOS app; on Windows you register it once with fused-render-open --register.

4Conditional templates

Each template can gate itself for a given file. Drop a condition.py with def main(path) -> bool next to the template — return True and it offers to render that file. Try it below: this condition only accepts files whose name contains special.

    return "special" in path.lower()

5Share templates — export & import

Templates you write are just folders — so they travel. Export any set of templates as a zip and hand it to a friend or teammate; they import it through a short wizard and get the same templates, with your file-type bindings offered as one-click suggestions.

Exporting

Open the Templates page and switch to the Library tab — it lists every template on your machine, core and custom. Select one or more and hit Export; a zip downloads with each template's folder at its top level.

What's in the zip

Template folders only — your registry (which file types open with what) never ships, because bindings are a local choice. Instead the zip carries a small recommendation.json sidecar recording which file types each template was bound to at export time, so the importer can offer those as suggestions rather than impose them.

Core templates export too

You can export built-in templates as well — handy as a starting point for a custom one, or to share a core template with someone on an older version.

Importing

On the receiving machine, the same Library tab has an Import button. Drop the zip in and a three-step wizard takes over:

1 · Review what's inside

The zip is unpacked to a staging area first — nothing touches your templates yet. You see each template it contains, whether it's valid, and whether it conflicts with a template you already have.

2 · Resolve conflicts & pick bindings

For each name clash choose overwrite, skip, or keep both (the incoming copy is renamed, and any bindings follow the new name). Below that, the exporter's recommended file-type bindings appear as toggle chips — new bindings default on; bindings for file types you've deliberately disabled default off and only re-enable if you opt in.

3 · Done

A summary shows what was imported, skipped, overwritten, or renamed — and which bindings were applied.

Bindings are additive

Applying a recommended binding appends the template to that file type's list — it never replaces your existing defaults. Reorder anytime on the File bindings tab.

Imports are safe by construction: zips are size- and path-checked (no zip-slip, no zip bombs), and until you commit the wizard, nothing is written to ~/.fused-render/templates/.

The app

Using Fused Render

The Fused Render window is a file explorer for your project folders — click an HTML file and it renders live. Here's how to get around it.

Bookmarks

Pin any page you're working on so it's one click away — the + Bookmark button, top right. Bookmarks live in the left sidebar (and group into folders); each stores the file and its query params, so you return to the exact state.

Split view

Open two files side by side — a page and its Python, or two previews — and work across them without losing either. Each pane keeps its own URL and state.

Opening files with templates

Any file on your machine can render through a template — right-click it and pick Open with FusedRender (registered by the desktop app). See Templates for how templates are chosen and customized.

Data

Mount

Mount a remote store — S3-compatible buckets, Google Drive, anything rclone speaks — and it shows up in the file tree like a local folder, ready to browse and open through a template.

1Mounting a remote

Mounts are powered by rclone, so anything rclone speaks works. The Add mount form asks for just three things — a name, a remote, and a path. Public S3 works with no setup; other backends (GCS, Google Drive) need a one-time rclone config in a terminal, after which they appear in the Remote dropdown. Below are the exact values to fill in for each.

🪣 Amazon S3 — public bucket In-app, no setup

Fused Render ships a built-in public-S3 remote (no credentials), so open data is a copy-paste. Here we mount building footprints from source.coop:

source_coop
AWS S3 — public buckets (no credentials) ▾
us-west-2.opendata.source.coop/vida/google-microsoft-open-buildings/pmtiles/by_country/country_iso=AND
Region us-west-2 is set by that remote already. For a private bucket, pick the credentialed S3 remote instead and it'll ask for your access keys.
☁️ Google Cloud Storage rclone config first

GCS needs a one-time remote. In a terminal run rclone config, add a remote of type google cloud storage, and name it gcs (for public buckets, answer yes to anonymous access; otherwise point it at a service-account key):

rclone config          # New remote → name: gcs → storage: google cloud storage

Then in the Add mount form, the gcs remote is now in the dropdown:

gcs_landsat
gcs ▾
gcp-public-data-landsat/LC08/01/044/034
Path is bucket/prefix. gcp-public-data-landsat is an open Google bucket — no credentials needed with anonymous access.
📁 Google Drive rclone config first

Google Drive is an OAuth backend, so the browser sign-in happens during rclone config. Add a remote of type drive, name it gdrive, and complete the sign-in it opens:

rclone config          # New remote → name: gdrive → storage: drive → sign in when the browser opens

Then mount a folder from your Drive by name (leave the path blank for the whole Drive):

my_drive
gdrive ▾
Reports/2026
Path is a folder inside your Drive (e.g. Reports/2026), or empty for the root.

2Navigate & open

Once mounted, source_coop appears in the file tree beside your local folders. Open it, click AND.pmtiles, and it renders through the pmtiles template as an interactive map of Andorra's buildings — nothing downloaded up front, just the byte ranges the map needs.

  1. Add the mount with the values above.
  2. In the file tree, expand source_coop → the country_iso=AND folder.
  3. Click AND.pmtiles → it opens as a map.

Publish

Deploy your Render · Beta

Take a project that runs on your machine and deploy it so others can open it in a browser.

Beta · limited availability

Performance won't match local, and you can't move much data through a deployed app yet. Expect rough edges while this matures.

1Requirements

  • Turn Deploy on — it's off by default while in beta. Open Preferences and enable the Deploy button; it then appears on every page.
  • A Fused account — sign up at fused.io and sign in with the fused CLI.
  • Your project should already run locally the way you want it.
Enable it first

Don't see a Deploy button on your pages? It's hidden until you switch it on in Preferences — that's expected during the beta.

2Deploying

Hit Deploy on any page. Pick which files ship (the page, its Python, and any bundled data), choose a managed Fused environment, and you get a shareable open.fused.io URL — redeploying keeps the same link.

3See a live one

The COG Overview Pyramid example, deployed — it opens a cloud-optimized GeoTIFF and stacks every resolution level into an interactive 3D pyramid, with each level's pixel size and byte cost. Open it in your browser, no install:
