# fused-render — Decision Log & Project Context

Purpose: this file + `SPEC.md` + `ARCHITECTURE.md` carry the *entire* design conversation.
A fresh session should be able to continue the project from these three files alone.

---

## What we are building (one paragraph)

A fully local file explorer: a Python server on `127.0.0.1` + a browser UI to browse the **entire computer's** filesystem and preview files. The core primitive is **renderable HTML**: any `.html` file previews live and gets an injected JS runtime with (a) `fused.runPython(pyPath, params)` — executes the `main()` function of a local Python file and returns its JSON result — and (b) `fused.params` — string key/values synced into the browser URL, so any view's state is bookmarkable/refreshable. **Preview templates** (parquet table, image, text …) are just renderable HTML files shipped inside the app that receive the target file as a reserved `_file` param — the same two primitives are the entire rendering power of the system. Local only, forever; never cloud — reframed, not overturned, by mounts (D102): the app itself still only ever reads local absolute paths; remote storage enters exclusively as an rclone-managed *mount* presenting as a local folder, so nothing downstream of the mount layer speaks a cloud protocol.

### Editing decisions (2026-07-05)

| # | Decision | Choice | Rationale / rejected alternatives |
|---|---|---|---|
| D36 | Cross-origin POST guard | The two mutating/executing endpoints (`POST /api/run`, `POST /api/fs/write`) require a custom `X-Fused: 1` request header; missing/wrong → 403 | Read endpoints are already safe cross-origin (the browser blocks a foreign page from reading our response), but a POST can be fired blind via a no-cors fetch by any website open in the same browser. A custom header forces a CORS preflight, which fails cross-origin since we send no CORS headers, so only our own same-origin pages get through. **NOT authentication** (D3 stands — no tokens, no user accounts): it only blocks blind cross-origin POSTs; anything that can already run our JS is unaffected. Cheap and removable if the descoped security layer lands. |
| D37 | File editing | `POST /api/fs/write` (atomic temp-file + `os.replace`, preserves mode) with optimistic locking on `st_mtime`; `code_template.html` is now an always-editable CodeMirror buffer (Save / Cmd+S, conflict banner offering Reload/Overwrite) | Supersedes SPEC §1 non-goal "no editing in v1". Optimistic lock (409 on stale/deleted mtime) chosen over no-lock (silent clobber) and over server-side locking (needs state, D-against SV-1). HTML "Source" view reuses the same editable template, so HTML editing came free |
| D38 | Bookmark updating | Opening a bookmark **via sidebar click** arms it (sessionStorage `fused.armedBookmark`); when view params diverge from the saved url on the same page, an "Update bookmark" button appears left of "+ Bookmark" and overwrites the bookmark's url with the current one. Changing pathname disarms **permanently** — only a new sidebar click re-arms (owner-chosen over re-arm-on-return). Shell observes param changes by wrapping `history.replaceState` (the iframe runtime writes params through it; no native event exists) | Keeps D20 verbatim-URL model — update = overwrite url, no structured diffing. sessionStorage so tracking survives refresh but not new tabs |

### Layout URL grammar — parenthesized `_layout` (2026-07-06, spec = SPEC §14.1 LM-2)

| # | Decision | Choice | Rationale / rejected alternatives |
|---|---|---|---|
| D51 | `_layout` scope grammar | The entire `_layout` value is **parenthesized and emitted last**: `?global=1&_layout=(/a.html?x=1&y=2,/b.html?z=3)`. Parens delimit param scope visually (inside = iframe-local, outside = global) and structurally: **`&` is literal inside them** — segment queries read exactly as written, no `%26` soup. Wrap is codec-transparent (`parseLayout` already reads `(A,B)` as `A,B`); only the URL layer changed. All shell-query reads go through `layout-codec.js` `splitShellSearch` (balanced-paren scan; safe — literal parens inside segments are codec-escaped, so span parens are structural and balanced): panel.js, tabs.js, breadcrumb's bookmark-equality `sameSearch`. The runtime (standalone, no imports) duplicates the scan as `splitSearch`: get/getAll parse only the non-layout remainder; `set()` reinserts the raw span untouched and last — also fixes the old wart where `URLSearchParams.toString()` percent-mangled a readable `_layout` on the first `fused.params.set()`. **Strict read** (owner call): unwrapped `_layout` = not this grammar, reads as absent, key dropped on next sync — old-format bookmarks fall back to a single start-dir pane/tab. **Paste breakage accepted** (owner call): auto-linkers may eat the trailing `)`; unbalanced span = invalid layout (span still excluded from params), missing-layout fallback | Chosen over "enforce `_layout` last as position-only convention" (look without guarantee; `&` stays escaped) and over "raw tail after `_layout=`" (position-dependent; trailing hand-added `&debug=1` silently swallowed into last segment — parens make position a convention and the boundary explicit, params after `)` stay global). Lenient read of old grammar rejected (owner call): one grammar, no two-format parse paths. Auto-closing missing trailing parens rejected with it: same leniency, masks truncation |

### Explorer search — streamed breadth-first walk (2026-07-09, M14)

| # | Decision | Choice | Rationale / rejected alternatives |
|---|---|---|---|
| D85 | The explorer search walks breadth-first, streams NDJSON, and scores client-side; the entry cap is a safety valve, not a budget | `/api/fs/walk` rewritten from `os.walk` (depth-first) to a FIFO `_walk_bfs` generator: **every depth-N entry emits before any depth-N+1**, so an early stop always leaves complete shallow coverage. `?stream=1` streams `{"entries":[…]}` NDJSON batches (500/line) + one terminal `{done, truncated, total}` line; client disconnect closes the generator and cancels the walk. Non-stream shape unchanged for old clients. Pruning: `.git`/`.venv` join `WALK_IGNORE_DIRS` (never emitted nor descended, hidden mode or not); macOS packages (`.app`/`.framework`/`.bundle`/`.photoslibrary`) emit as one leaf entry, never descended (Finder semantics). Cap raised 20k → **200k** and reframed as a memory/latency valve — BFS means it only ever cuts the deepest levels of pathological trees, and the UI reports it (live `N matches · M scanned…` counter, `+`-suffixed final count, zero-match message naming the covered count). The shell fetches **one hidden-inclusive dataset** (`hidden=1` always) and gates dot-entries at display time by query intent (dot-leading segment shows them) — `.py` is an extension search, intent flips never refetch. Scoring stays client-side, incremental: flushes throttled to 200 ms, each scoring only newly arrived entries merged into the prior ranked list; full re-scan only on query/intent change. Results page in 250-row chunks via a sentinel row + IntersectionObserver; the query mirrors to the URL debounced 200 ms (Safari throttles `replaceState` and then throws). | Owner report 2026-07-09: search unusable — depth-first + 20k cap let `~/Desktop` eat 15,926 slots so Movies/Music/Pictures had **zero** searchable children; `/Applications` burned 7,557 entries inside one `.app`; first paint blocked ~1 s on the full walk; a later report found per-chunk full re-scoring saturated the main thread (stuck stale-dim, dead clicks) — hence the throttle + incremental merge. Client-side scoring kept over a server-side paged search (owner probed this): ranked fuzzy search must scan the full corpus per query wherever it lives, the "server" is the same machine (moving ~30 MB between processes saves nothing), per-keystroke ranking then costs an HTTP round trip + Python-speed matching, and the local-tool precedent (fzf, VS Code Quick Open, Everything) is corpus-in-matcher-memory. Revisit only as a persistent Spotlight-style index if the cap ever genuinely bites. Rejected: walk-more-on-scroll (can't rank what you haven't walked — scroll pages the *rendered* rows instead); a second hidden-only walk per dot-query (double fetch, wrong-dataset races — the old `forHidden` state machine dissolved once one dataset carried both). |

### Explorer search — gitignore-driven pruning (2026-07-10)

| # | Decision | Choice | Rationale / rejected alternatives |
|---|---|---|---|
| D100 | The walk prunes what each repo's own gitignore ignores, via a streaming `check-ignore` co-process; the hardcoded dir set shrinks to a universal floor; search excludes rather than dims (revises SR-2, D85's pruning + D-#75's walk dimming) | Inside a git repository, `_walk_bfs` drops every entry the repo's gitignore rules ignore — not emitted, not descended. Verdicts come from `_IgnoreOracle`: one long-lived `git check-ignore --stdin -z -v -n` subprocess per repo answering path queries incrementally (measured ~14 µs/query; `-v -n` echoes all four NUL fields for every query, making the stream pairable, and exposes the matched pattern's sign so negations like `!keep.log` are honored — a match on a `!` pattern is NOT ignored). Queries are chunked (200/write) so git's stdout can't fill the pipe mid-write (co-process deadlock). Repo tracking rides the BFS queue: each dir inherits its parent's repo root, a `.git` child (dir or worktree/submodule gitfile) starts a nested repo with its own rules, and a walk rooted below a repo top resolves it via one `git rev-parse --show-toplevel`. A `.gitignore` with NO repo in scope (un-inited project, Obsidian vault — the owner's content-engine/.env report) starts a standalone ignore root: the oracle grafts the directory onto a shared lazily-created empty `GIT_DIR` as its `GIT_WORK_TREE`, so the same check-ignore co-process honors standalone .gitignore files, cascading and negations included. Oracles are LRU-capped (`WALK_MAX_ORACLES` = 8) and all closed when the walk generator ends; any git failure marks that oracle broken → no pruning for that repo (optimization, never a dependency). `WALK_IGNORE_DIRS` shrinks in role to the universal floor (`node_modules`, `__pycache__`, `venv`, `.venv`, `.git`, `site-packages` — new) covering non-repo junk and `.git` itself. Consequence: walk entries no longer carry the `ignored` dimming flag (#75's per-batch `_annotate_ignored` removed — nothing ignored survives to dim); `/api/fs/list` keeps dimming. Measured: fused-render repo walk 292 entries in 43 ms (gitignored `shell-dist`, lockfile churn etc. gone). | Owner call 2026-07-10 ("check if the folder has .gitignore and ignore those files — makes much more sense than hardcoding"): the hardcoded set could never enumerate every ecosystem's junk (`dist`, `build`, `.next`, `target`, `.turbo`, …), while each repo's gitignore is the project's own authoritative noise list. Search-excludes-vs-listing-dims mirrors VS Code (explorer shows gitignored, Quick Open hides). Rejected: parsing .gitignore in Python (reimplementing git's semantics — nested files, excludesfile, negation precedence — when `check-ignore` is the authority, same posture as #75's `_git_ignored`); one subprocess per directory (thousands of ~10 ms spawns per walk); one giant upfront batch per repo (the walk discovers paths incrementally — a co-process answers as it goes); pruning by the `ignored` flag client-side (entries would still be walked, streamed, and scored — the win is not descending at all). No user toggle yet — add "search ignored files too" only if someone actually needs to find a build artifact. |
